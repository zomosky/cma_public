# ARCHITECTURE

## 1. 目标与边界
- 本项目是“天气信息采集 + 结构化汇总 + Markdown 输出”的自动化任务流程。
- 当前阶段只定义可长期维护的架构约束，不在本文件中放置业务实现细节。
- 业务规则以 `PROJECT.md` 为准；本文件只约束技术方向与模块边界。

## 2. 建议技术栈（保守）
- 语言：Python 3.11（可向下兼容 3.10）。
- 环境管理：Conda（推荐）或 venv。
- 依赖管理：`requirements.txt`（稳定、易部署）。
- 核心能力：
  - HTTP 抓取：`requests`
  - 解析：`beautifulsoup4` / `lxml`
  - OCR：后续按稳定方案接入（接口抽象，不与实现绑定）
  - 大模型调用：通过统一 Client 适配层接入
- 调度：优先系统级定时（cron），避免引入重型编排系统。

## 3. 目录与模块约束
- `PROJECT.md`：需求与业务规则来源。
- `ARCHITECTURE.md`：技术栈与架构约束。
- `RULES.md`：协作与变更规则。
- `SESSION_STATE.md`：当前会话状态。
- `DECISIONS.md`：架构决策记录。

后续新增代码时建议：
- `src/collectors/`：数据采集
- `src/parsers/`：结构化解析
- `src/services/`：模型与外部服务调用
- `src/pipeline/`：任务编排
- `tests/`：测试代码

## 4. 架构原则
- 分层清晰：采集、解析、总结、输出解耦。
- 配置外置：密钥与环境参数不硬编码，不提交到仓库。
- 可观测性：关键步骤必须有日志与错误上下文。
- 幂等性：同一日期重复执行时结果可预测。
- 可替换性：第三方接口通过适配层隔离。

## 5. 非功能约束
- 稳定优先于功能堆叠。
- 默认单机可运行，先满足可复现再做扩展。
- 任何新增技术选型必须先记录到 `DECISIONS.md`。
